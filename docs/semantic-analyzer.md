# Semantic Analyzer

## Purpose

Identifies high-level UI components from HTML structure, class names, and computed styles. This enables the generators to emit platform-native components (e.g., `Card`, `AppBar`, `NavigationBar`) instead of generic containers.

## Implementation

**File:** `packages/semantic-analyzer/index.ts`  
**AI variant:** `packages/semantic-analyzer/ai.ts`  
**Package:** `@html-native/semantic-analyzer`

## Exported Functions

### `detectSemantics(styledNodes: StyledNode[]): SemanticHint[]`

Runs rule-based detection on a styled node tree.

**Parameters:**
- `styledNodes` — Array of StyledNodes (output of `applyStyles()`)

**Returns:** `SemanticHint[]` — ordered by discovery (no sort needed).

### `createAiDetector(config?: AiDetectorConfig): SemanticDetector`

Creates an AI-powered detector that calls Ollama for enhanced semantic detection. Falls back to rule-based detection on any error.

**Parameters:**
- `config.model` — Ollama model name (default: `qwen2.5:7b`)
- `config.baseUrl` — Ollama server URL (default: `http://localhost:11434`)
- `config.timeout` — Request timeout in ms (default: 30000)

**Returns:** A `SemanticDetector` function: `(nodes: StyledNode[]) => Promise<SemanticHint[]>`

## Detection Rules

### Navigation Bar
- **Trigger:** `tag === 'nav'` or class contains `nav`, `navbar`, or `navigation`
- **Confidence:** 0.9
- **Reason:** `'nav tag or class'`

### Card
- **Trigger:** class contains `card`, `card-body`, or `card-content`
- **Confidence:** 0.85
- **Fallback:** `div` with both `box-shadow` and `border-radius` styles
- **Fallback confidence:** 0.6

### Hero Section
- **Trigger:** class contains `hero`, `hero-section`, or `banner`
- **Confidence:** 0.9

### Dialog/Modal
- **Trigger:** class contains `modal`, `dialog`, or `overlay`
- **Confidence:** 0.9

### AppBar
- **Trigger:** `tag === 'header'` AND class contains `app-bar`, `toolbar`, or `topbar`
- **Confidence:** 0.85

### Sidebar
- **Trigger:** class contains `sidebar`, `side-bar`, or `drawer`
- **Confidence:** 0.9

### Footer
- **Trigger:** `tag === 'footer'` or class contains `footer`
- **Confidence:** 0.9

### Tabs
- **Trigger:** class contains `tabs`, `tab-bar`, or `tab-container`
- **Confidence:** 0.85

### Form
- **Trigger:** `tag === 'form'`
- **Confidence:** 0.9

## AI Enhancement (Optional)

When `--ai-enhance` is passed to the CLI, the pipeline calls Ollama (a local LLM server) for additional semantic hints.

### Flow

```
StyledNode[]
    │
    ├──▶ rule-based detector (always runs)
    │
    └──▶ AI detector (if enabled)
              │
              ├──▶ Ollama available ──▶ parse AI response
              │
              └──▶ Ollama unavailable ──▶ fallback (empty AI hints)
                          
    └──▶ Merge: AI hints + rule hints
         Sort by confidence
         Deduplicate by type + nodeId
```

### Merge Strategy

1. AI hints and rule-based hints are combined
2. Sorted by confidence (highest first)
3. Deduplicated by `type + node.nodeId` — higher-confidence hint wins

### Prompt Engineering

The AI prompt explicitly instructs Ollama to:
- Return JSON only (no markdown, no explanation)
- Detect: cards, navigation, forms, hero sections, dialogs, reusable components
- **Never generate platform code** (Flutter, SwiftUI, Compose)

## Core Types

```typescript
interface SemanticHint {
  type: UiNodeType;       // The detected component type
  confidence: number;      // 0.0 to 1.0
  node: HtmlNode;          // The matched HTML node
  reason: string;          // Human-readable explanation
}

type SemanticDetector = (nodes: StyledNode[]) => Promise<SemanticHint[]>;
```

## Testing

The semantic analyzer has 3 dedicated tests covering:
- Fallback to rule-based detector when Ollama is unavailable
- Synchronous pipeline operation without AI flag
- Platform code never generated by the AI module
